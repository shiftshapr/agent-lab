# BRC333 Monument Protocol — Future-Proof Satplications

This briefing defines a **Monument Protocol** for BRC333-oriented satplications.

The purpose of the protocol is to make satplications more:

- durable
- referenceable
- governable
- discoverable
- bridgeable into future interface layers

This is about making culturally or civically important satplications **future-proof**.

The protocol should allow a satplication to function not just as an inscription or application, but as a **monument-class knowledge object** that can persist across interface eras.

## Program mapping: Civic Mason and Pacha badges

- **Civic Mason** digital badges are issued on **BRC333**: satplications (and, where warranted, **monument-class** structures as defined in this document) so credentials stay **durable, referenceable, stewarded, and bridgeable** across viewers and future layers.
- **Pacha** digital badges follow a **similar** BRC333-oriented approach—same family of **identity, persistence, and discoverability**—with **kid-specific** policy (safety, privacy, moderation, parent-paid commerce) spelled out in the appendix below.

---

# 1. Core Idea

A **monument** is a digitally recognized structure of enduring significance.

In BRC333 terms, a monument may be:

- a satplication
- a canonical inscription set
- a knowledge interface
- a cultural archive
- a civic coordination object
- a protocol reference point

A monument is not defined only by storage.
It is defined by:

- persistence
- identity
- referenceability
- stewardship
- social and protocol recognition

---

# 2. Why a Monument Protocol Is Needed

Satplications can be powerful but fragile if they depend too heavily on a single interface, host, or interpretation layer.

A Monument Protocol helps ensure that satplications remain:

- intelligible over time
- re-renderable in future clients
- referenceable by other systems
- preservable across shifts in interface standards

The protocol should support the idea that:

> a satplication can outlive the interface that first displayed it.

---

# 3. Design Goals

The Monument Protocol should make satplications:

## A. Durable
Able to persist as meaningful structures over long periods.

## B. Portable
Able to be interpreted in future viewers, clients, and meta-layers.

## C. Governable
Able to be stewarded, annotated, amended, or referenced through explicit processes.

## D. Discoverable
Able to be located, indexed, and understood as a monument-class object.

## E. Bridgeable
Able to connect to future systems such as Overweb / Canopi-style bridge layers.

---

# 4. What Makes a Satplication “Monument-Class”

A satplication becomes monument-class when it has:

- a durable identity
- a known canonical reference
- explicit metadata
- stewardship context
- a path for future interpretation

This does not require centralization.
It requires a **protocol-level structure** for meaning and continuity.

---

# 5. Monument Components

Each monument-class satplication should include or be associated with the following:

## A. Canonical Identifier

A stable identifier for the monument.

This may include:

- inscription reference(s)
- collection reference
- canonical protocol ID
- human-readable monument ID

## B. Metadata Layer

Minimum metadata should include:

- title
- summary
- creator / initiator
- creation date
- protocol version
- inscription references
- subject / theme
- intended function

## C. Stewardship Layer

A monument should indicate:

- who stewards it
- whether stewardship is individual, communal, or protocol-level
- how future updates or interpretations are handled

## D. Rendering / Interpretation Guidance

A satplication should not rely only on one viewer.

It should include enough structure so future systems can:

- reconstruct it
- render it
- interpret its logic
- understand its significance

## E. Relationship Layer

A monument should be able to relate to:

- other monuments
- inscriptions
- artifacts
- governance systems
- external resources

---

# 6. Protocol Layers

The Monument Protocol should be thought of in layers.

## Layer 1 — Storage / Inscription Layer

The underlying durable inscription or inscription set.

This is the persistence base.

## Layer 2 — Metadata Layer

Human and machine-readable metadata describing what the monument is.

## Layer 3 — Interpretation Layer

Rules or guidance for how clients should understand / render it.

## Layer 4 — Relationship Layer

Typed links connecting the monument to:

- related satplications
- historical context
- successor systems
- artifacts
- bridge systems

## Layer 5 — Stewardship / Governance Layer

Defines how the monument is maintained, referenced, or extended over time.

---

# 7. Future-Proofing Principles

To future-proof satplications, the protocol should follow these principles:

## A. Separate content from interface

The satplication must not depend entirely on one rendering surface.

## B. Preserve enough metadata for re-interpretation

Future viewers should be able to reconstruct intent and structure.

## C. Allow bridgeability

The monument should be linkable into later systems such as:

- BRC333 viewers
- future satplication browsers
- Overweb applications
- Canopi-like bridge layers

## D. Keep identifiers stable

References to the monument should not depend on unstable UI paths.

## E. Preserve lineage

If monuments are updated, forked, or reinterpreted, that lineage should be explicit.

---

# 8. Monument Relationships

The protocol should support typed relationships such as:

- references
- derived_from
- fork_of
- commemorates
- extends
- supersedes
- anchored_by
- rendered_by
- interpreted_by
- bridged_to

This allows monument-class satplications to become part of a broader knowledge graph.

---

# 9. Stewardship Model

A monument should have a visible stewardship model.

Possible models:

- creator-steered
- guild-steered
- layer-steered
- protocol-steered
- community-maintained

The stewardship model should clarify:

- who may update metadata
- who may register interpretations
- who may publish bridge mappings
- how disputes are handled

---

# 10. Versioning and Lineage

Future-proof satplications need explicit versioning.

A monument protocol should support:

- original version
- amended versions
- forked versions
- successor monuments

This allows future systems to understand whether something is:

- canonical
- deprecated
- forked
- superseded

---

# 11. Rendering Independence

The protocol should assume that any current UI may disappear.

Therefore monument-class satplications should preserve enough structure for:

- future reconstruction
- alternate visualizations
- machine-readable interpretation
- migration to new interaction layers

This is what makes them future-proof.

---

# 12. Relationship to Satplications

Not every satplication needs to be a monument.

But the protocol should allow satplications to become monument-class when they are:

- culturally significant
- civically significant
- protocol-significant
- historically important
- canonical for a domain

This creates a category above ordinary satplication instances.

---

# 13. Relationship to Overweb / Future Interface Layers

The Monument Protocol should anticipate future interface layers.

This means monuments should be bridgeable into:

- browser overlays
- context layers
- annotation systems
- governance hubs
- future Canopi / Overweb environments

The monument should remain meaningful even if its original interface no longer exists.

---

# 14. Minimal Protocol Requirements

A minimal monument registration or declaration should include:

- canonical ID
- title
- summary
- inscription or content reference
- protocol version
- stewardship model
- relationship capacity
- rendering guidance pointer

This is the minimum viable future-proofing layer.

---

# 15. Why This Matters

BRC333 is about more than storing things.
It is about enabling enduring, dynamic, meaningful digital structures.

A Monument Protocol helps ensure that important satplications are not lost to:

- interface decay
- client fragmentation
- metadata loss
- context collapse
- cultural forgetfulness

It makes satplications legible to the future.

---

# 16. Summary

The BRC333 Monument Protocol should make future-proof satplications:

- persistently identifiable
- richly described
- bridgeable into future systems
- stewarded over time
- structurally interpretable beyond their original UI

In short:

> A monument-class satplication is a durable digital structure whose meaning, identity, and relationships survive beyond the interface that first revealed it.

That is the standard the protocol should aim for.

# Appendix: Pacha digital badges — commerce and access

This section records how **Pacha digital badges** are funded and customized. **Technically**, Pacha badges are implemented in the **same BRC333-oriented family** as Civic Mason (see *Program mapping* above); **policy** here is specific to the kid-facing layer. Civic Mason’s teen/adult program has its own badge rules and stewardship, still on BRC333.

## Payer of record

- **Parents or guardians** are the paying party for minors.
- Purchases and subscriptions should be clearly labeled for **adult approval** at checkout and in-account settings.

## Pricing model

### Base charge

- A **base fee** covers core participation in the Pacha badge system: earning, displaying, and progressing through the canonical badge path (exact inclusions are defined in product policy).
- The base may be offered as a **one-time**, **term**, or **subscription** depending on the product; the principle is that **core badges are not paywalled behind endless upsells**—families pay once for the baseline experience.

### Accessorize and express

- Families may **pay to accessorize**: optional cosmetics, frames, themes, or flair that sit **on top of** earned badges.
- Kids can add **creative expression** (within moderation and safety rules) where the product allows it; paid tiers may unlock additional expressive tools or slots.
- **Artifacts** may be **attached** to badges or profiles (e.g. drawings, voice notes, short captions, scanned art, links to approved projects)—with clear rules for **what counts as an artifact**, **size limits**, and **removal/moderation**.

### Geography: sliding scale

- Pricing uses a **sliding scale by geography** (region or country) so the base charge and add-ons align with **local purchasing power** and partner economics.
- The same **ethical bar** applies everywhere: no predatory dark patterns, clear disclosure of what is included in base vs paid extras, and **access programs** (free or subsidized distribution) remain compatible with the mandate to reach the most widely read children’s book and an active global community.

## Relationship to Civic Mason

- **Overlap is allowed** and should **vary by kid** (e.g. some continue Pacha-only longer; some begin Civic Mason–adjacent activities in ninth grade per program rules).
- **Civic Mason** badges: **BRC333** issuance per this protocol; commerce and governance for that program are documented separately from this appendix.
- **Pacha** badges: **similar BRC333-style** issuance; this appendix governs **parent-paid** and **geography** rules for the kid layer unless explicitly bridged in policy.
